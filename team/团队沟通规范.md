# 团队沟通规范

> OpenClaw 项目交付团队 - 沟通机制与记录

---

## 💬 沟通原则

### 1. 唯一对接原则

**小白是唯一与大哥直接沟通的角色**

```
大哥
  │
  ▼
小白（项目经理）
  │
  ├─ 分配给团队成员
  ├─ 跟踪执行进度
  └─ 汇总汇报大哥
```

**禁止**: 其他成员直接与大哥沟通
**例外**: 紧急情况，小白授权后可直接沟通

### 2. 分层沟通原则

| 层级 | 沟通方式 | 示例 |
|------|---------|------|
| **战略层** | 小白 ↔ 大哥 | 需求确认、项目汇报 |
| **战术层** | 小白 ↔ 团队成员 | 任务分配、进度跟踪 |
| **执行层** | 团队成员之间 | 技术对接、问题协作 |

### 3. 沟通时效

| 优先级 | 响应时间 | 场景 |
|--------|---------|------|
| **P0-紧急** | 15分钟内 | 系统故障、安全事件 |
| **P1-高** | 1小时内 | 阻塞性问题 |
| **P2-中** | 4小时内 | 一般问题 |
| **P3-低** | 24小时内 | 优化建议 |

## 🔄 标准沟通流程

### 场景1：需求确认

```
【发起】大哥（飞书）
    "需要开发一个数据可视化功能"
    
【接收】小白（AITest01）
    时间：2026-03-19 09:00
    动作：记录需求，分析理解
    
【确认】小白 → 大哥（飞书）
    "收到。需求理解如下：
     1. 数据可视化功能
     2. 支持实时更新
     3. 预计3天完成
     请确认理解是否正确？"
    
【确认】大哥（飞书）
    "确认，正确"
    
【记录】小白
    位置：team/communications/2026-03-19/需求确认-数据可视化.md
```

### 场景2：任务分配

```
【分配】小白（内部）
    时间：2026-03-19 09:30
    动作：唤醒小红，需求分析
    
【执行】小红（AITest01）
    激活：✅ 需求分析
    输出：需求文档
    休眠：⏸️
    
【分配】小白 → 小灰（SSH）
    命令：ssh 腾讯OpenClaw "技术方案设计"
    
【执行】小灰（腾讯OpenClaw）
    输出：技术方案.md
    
【记录】小白
    位置：team/communications/2026-03-19/任务分配-数据可视化.md
```

### 场景3：进度汇报

```
【定时】小青（AITest01）
    时间：每天 16:00
    动作：生成进度报告
    
【汇总】小白（AITest01）
    动作：汇总各成员进度
    
【汇报】小白 → 大哥（飞书）
    "今日进度：
     - 小灰：技术方案完成 ✅
     - 小蓝：开发中，60% ⏳
     - 小黑：等待开发完成 ⏸️
     预计明日完成开发"
    
【记录】小白
    位置：team/communications/2026-03-19/进度汇报.md
```

### 场景4：问题升级

```
【发现】小蓝（腾讯OpenClaw）
    问题：数据库连接失败
    尝试：重启服务，无效
    
【求助】小蓝 → 小灰（SSH）
    沟通：技术讨论
    结果：无法解决
    
【升级】小灰 → 小白（SSH返回）
    汇报：问题描述 + 已尝试方案
    
【决策】小白（AITest01）
    评估：影响范围、紧急程度
    决策：P1-高，1小时内解决
    
【沟通】小白 → 大哥（飞书，如需要）
    "遇到技术问题，预计延迟2小时，是否接受？"
    
【解决】小白协调资源
    动作：分配小灰专项解决
    结果：问题解决 ✅
    
【记录】小白
    位置：team/communications/2026-03-19/问题升级-数据库连接.md
```
